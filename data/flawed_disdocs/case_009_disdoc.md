### 1. Problem definition

#### several Origin

The core problem is to improve engineering efficiency and productivity across Swiggy's engineering vertical. The initiative, driven by the Developer Experience (DXP) team, involves the adoption and application of Generative AI (GenAI) based Code Generation (CodeGen) tools to accelerate various phases of the Software Development Life Cycle (SDLC). The adoption started in March several and has reached over several of the engineering organization.

#### several Relevance & reasons

Several engineering tasks were identified as time-consuming, repetitive, or having room for quality improvement. The GenAI tools are intended to address these specific pain points:

*   **Unit Test (UT) Development**: Writing extensive test cases is essential for code quality but is a time-consuming process for developers.
*   **Application Size Optimization**: A key objective for the Delivery Experience (DE) app is to streamline the application by removing redundant and outdated code and assets to reduce the overall APK size.
*   **Repetitive Data Gathering**: Developers manually gathered data from multiple sources (Google Play Store, Firebase Crashlytics, Sentry, PagerDuty, NewRelic, UserExperior) to create decks for the recurring Technical Review Meeting (TRM), a cumbersome and repetitive task.
*   **Urgent Production Fixes**: Production incidents sometimes require specialized, one-off scripts to be developed under time pressure to update specific records.
*   **Code Review Overhead**: Manual Pull Request (PR) reviews are a crucial but potentially time-intensive part of the SDLC.

#### several Expectations

The primary expectation is to improve developer efficiency and accelerate delivery. Specific expectations for different use cases include:
*   Significant time savings in development tasks.
*   Improved code quality by catching bugs early and ensuring better structure.
*   Reduction in application bundle size.
*   Automation of manual, repetitive reporting tasks.
*   Faster response times during production incidents.

#### several Previous work

The baselines for these initiatives were the existing manual processes:
*   Developers writing all unit tests manually.
*   Manual identification and removal of redundant application assets.
*   Developers manually collecting metrics from various dashboards for TRM decks.
*   Manual scripting or data manipulation for production hotfixes.
*   Fully manual code review processes by engineering peers and leads.

### 2. Goals and anti-goals

#### 2.1. Goals

*   **Primary Goal**: Improve overall engineering efficiency, in line with industry observations of ~15% improvement.
*   **Reduce UT Development Time**: Achieve a 60-70% reduction in time spent writing unit tests.
*   **Optimize Application Size**: Reduce the APK size of the Delivery Experience app. A concrete result was a 4.8 MB reduction.
*   **Automate Reporting**: Eliminate the manual effort required to build TRM decks by automating data gathering.
*   **Accelerate Incident Response**: Reduce the time taken to create scripts for urgent production data fixes. A specific instance saw a reduction from a potential 20 minutes to 5 minutes.
*   **Enhance Code Review**: Automate the detection of fundamental coding issues (e.g., error handling, Null Pointer Exceptions) and improve code documentation during the PR review process.

### 3. Risks and constraints

*   **Organizational Constraint**: Initial adoption was challenging due to a learning curve in effective prompt usage and a lack of common knowledge on best practices. This was mitigated by the DXP team through office hours, tech talks, and hackathons.
*   **Risk: Ineffective Measurement**: Difficulty in automatically measuring the true effectiveness of the tools. Current metrics like acceptance rate are considered insufficient. The desired metric is the percentage of total code in a PR contributed by GenAI.
*   **Risk: Poor Recommendation Quality**: The tools provide suboptimal recommendations for existing or unopened code within the IDE due to a lack of sufficient context. The DXP team is working with partners to solve this.
*   **Risk: Low Quality of Accepted Code**: There is a risk that the code suggested by GenAI and accepted by developers may not meet Swiggy's quality standards or coding guidelines.

### 4. Metrics and loss functions

#### 4.1. Offline metrics

*   **Acceptance Percentage**: The ratio of `lines of code accepted / lines of code suggested` by the CodeGen tool. This is a current effectiveness metric.
*   **Code Contribution Percentage**: A planned future metric to determine what percentage of the overall code in a raised or accepted PR was contributed by GenAI tools.
*   **Code Quality Score [inferred]**: To measure the quality of recommendations, the team is building an automated PR checker using GPT-4 based on Swiggy's coding guidelines. The output of this checker will serve as a quality metric. Currently, quality is assessed manually via reviews of sample PRs by tech leads.

#### 4.2. Online/business metrics

*   **Time Saved**:
    *   **Unit Tests**: 60–70% reduction in development time. A specific example cites 2 days of time saved for one module.
    *   **Production Incidents**: Time to create a recovery script reduced from 20 minutes to 5 minutes.
*   **Sprint-level Savings**: Effectiveness is measured by tracking savings at a Sprint level for selected projects.
*   **APK Size Reduction**: A direct measure of optimization success. A reduction of 4.8 MB (from 42.3 MB) was achieved for the DE app.
*   **Code Generation Contribution**: For the TRM Builder, ~80% of the code suggestions were derived from ChatGPT. For a specific UT module, 70% of the tests were tool-generated.

### 7. Baseline solution

The baseline for comparison is the fully manual process that existed prior to the adoption of GenAI tools for each use case:

*   **Unit Tests**: Developers writing all tests from scratch.
*   **Code Simplification**: Manual, ad-hoc efforts to find and remove redundant assets from the application repository.
*   **TRM Deck Automation**: Developers manually navigating to multiple dashboards (Google Play, Firebase, etc.) to collect data and build presentation decks.
*   **Data Recovery**: A developer manually writing a bash script or performing API calls, estimated to take 20 minutes for a specific incident.
*   **Code Review**: Engineers manually reviewing PRs for correctness, style, and potential bugs like NPEs and poor error handling.

### 8. Errors and their analysis

The article discusses challenges and contributions that highlight error types and analysis:

*   **Model Errors (Lack of Context)**: The primary source of error is the model producing poor or irrelevant recommendations when it lacks sufficient context about the existing codebase.
*   **Generated Code Quality Errors**: The generated code may not be of "good quality" and may not adhere to Swiggy's internal coding guidelines. This is a key concern the team is actively working to address.
*   **Analysis of Human-Written Code**: The GenAI code review tool is used to *analyze* human-written code and identify existing issues, such as:
    *   Inadequate error handling.
    *   Potential Null Pointer Exceptions (NPEs).
    *   Violations of basic programming language practices.
    *   Lack of sufficient code documentation or comments.

### 9. Training pipelines

Swiggy is utilizing third-party, pre-trained GenAI models like ChatGPT (including v4) and other CodeGen tools. The article does not mention any internal model training or fine-tuning. The focus is on the application and integration of these tools.

The "pipeline" is a developer workflow facilitated by the DXP team:
*   **Tooling**: ChatGPT, other unnamed CodeGen tools integrated into IDEs.
*   **Process**: Developers use these tools for specific tasks (e.g., UT generation, scripting) via prompts.
*   **Automation**: Scripts like the "TRM Builder" and the asset removal script were created with ChatGPT's help to automate specific workflows.
*   **CI/CD Integration**: A future goal is an "automated PR checker" built with GPT-4 that would integrate into the PR process to enforce coding guidelines.

### 10. Features

In the context of using LLMs, "features" correspond to the prompts and context provided to the model.

*   **Prompt Engineering**: The article emphasizes the importance of prompt quality and context. The DXP team conducted office hours and tech talks to train engineers on how to use prompts effectively.
*   **Incremental Prompts**: The TRM Builder was developed using "invaluable assistance through incremental prompts."
*   **Context Provisioning**: The code simplification script was significantly improved by "providing additional context through prompts" along with upgrading to ChatGPT v4.
*   **Contextual Challenge**: The biggest challenge identified is "Effective Context determination for existing code," highlighting that providing the right code context is the most critical "feature" for getting good results.

### 12. Integration and Serving

#### 12.1. API design

The system is a collection of tools and scripts integrated into the developer workflow, rather than a single serving endpoint.
*   **IDE Integration**: CodeGen tools are used for in-line code suggestions and comments directly within the developer's IDE.
*   **Chat Interface**: Developers use ChatGPT for interactive tasks like generating bash scripts or getting help with code.
*   **Scripting**: Custom scripts (e.g., TRM Builder, asset remover) are developed with GenAI assistance. These scripts in turn call APIs of other services (NewRelic, Firebase, etc.).
*   **PR Process Integration**: An automated PR checker is being developed to integrate into the SDLC at the code review stage.

#### 12.2. Infrastructure

The GenAI models are provided by third-party vendors (e.g., OpenAI's ChatGPT). Swiggy's DXP team manages the adoption, best practices, and integration of these external tools into internal developer workflows.

#### 12.3. SLAs, latency budgets, and fallback strategies

*   **Fallback Strategy**: The fallback for any failed or poor-quality generation is the original manual process: the developer can ignore the suggestion and write/review the code manually.
