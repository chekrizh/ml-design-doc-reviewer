Authors: Ozan Sonmez, Bjorn Burscher, Klaus Schaefers, Basak Eskili
At Booking.com, clear and timely communication between accommodation partners and guests is essential to a great travel experience. Guests often reach out with questions about check-in times, parking, or special requests, and partners usually reply manually through Booking.com’s messaging platform. To make this easier, we already offer response templates for common questions, but partners still need to search for and select the right one. During busy periods, this extra effort can slow things down, leaving travelers waiting for the reassurance they need before their trip. In some cases, delayed responses can lead to cancellations, resulting in lost bookings for partners.
To help with this, our team built a Generative AI (GenAI) agent that assists partners by automatically suggesting a relevant response to each guest inquiry. Depending on the message, it can surface an existing template or generate a tailored free-text answer, saving time and helping partners reply faster and more accurately.
Developing such an agent isn’t straightforward. We had to address inconsistent data quality across properties, gaps in response template coverage, and the challenge of supporting many languages. Accuracy and safety are critical: the system must avoid giving outdated or incorrect information, and some questions should never be answered automatically. By keeping humans in the loop, we can balance innovation with trust and reliability. On the technical side, the solution also needs to be fast, scalable, and cost-efficient.
This GenAI agent is just the beginning. It lays the groundwork for future automation that could make communication between guests and partners even smoother, so travelers can focus less on logistics and more on enjoying their journey.
Solution
We developed a tool-calling GenAI agent in Python, chosen for its rich ecosystem of libraries and frameworks for building intelligent systems. At the core, we use LangGraph, an open-source agentic framework that lets the agent reason about tasks and decide which tools to use to answer each guest inquiry.
Unlike a fixed, rule-based system, the agent works autonomously. It can call different tools to retrieve predefined response templates, access property or reservation data, and determine whether it can confidently generate a reply. Its goal is always to take one of three possible actions:
- Template Response: Use a predefined response template created by the partner. This is the preferred outcome, since it reflects the partner’s tone and style, ensuring consistency across communications.
- Custom Response: Generate a unique, free-text reply when no suitable template exists but enough contextual data is available to answer accurately.
- No Response: Refrain from answering when the system lacks the necessary information, or when the question falls into a restricted category (for example, refund requests).
To decide between these options, the agent follows a workflow. It first applies guardrail mechanisms to protect both partners and guests. It redacts any personally identifiable information (PII) from the incoming message and checks whether the topic belongs to a “do not answer” category. If the question passes these checks, an LLM helps determine which tools to use next.

Once the relevant tools are identified, the agent runs them concurrently, gathers their outputs, and uses the LLM again to reason over the results and generate a final response. This design carefully pre-selects tools based on the query and available context, avoiding unnecessary calls and keeping token usage and costs efficient. (See the diagram below for an overview.)
The agent runs as a microservice in our Kubernetes environment, allowing for seamless deployment and scalability. We use FastAPI as the web framework and OpenAI’s GPT-4 Mini as the LLM backend, accessed through our internal LLM gateway. The gateway adds critical safety layers such as prompt-injection detection before any data reaches the model. All of the agent’s tools are hosted on a central MCP server, also running on Kubernetes.
Tools
- Response Template Retrieval Tool: Helps the agent find response templates that best match a guest’s question in meaning. It embeds the message into a vector space and performs a k-nearest-neighbors search to retrieve the eight closest matches, filtering out weak ones with a similarity threshold. After testing several configurations, we selected MiniLM as the embedding model for its strong recall@k performance on real-world data. These embeddings are stored and indexed in Weaviate, our vector database, which provides fast semantic search. Template updates stream in real time via Kafka, ensuring the index always reflects the latest partner content.
- Property Details Tool: Retrieves specific property information by translating the agent’s requests into GraphQL queries against our backend systems.
- Reservation Details Tool: Provides reservation-level data such as room type, check-in and check-out dates, and number of travelers, giving the agent the context it needs to tailor replies accurately.

Evaluation
We followed a structured, data-driven evaluation process to ensure the AI agent delivers reliable and high-quality responses. Our approach combined manual review, automated evaluation, and operational monitoring to measure both performance and real-world impact.
To test our solution, we built a dataset of representative guest messages, each paired with real partner replies and contextual data such as property and reservation details. The dataset included both template-based and free-text responses, capturing the variety of real partner-guest communication.
Using the SuperAnnotate platform, we ran multiple rounds of manual annotation, reviewing the quality of AI-generated answers. Along the way, we conducted error analysis to identify recurring issues, such as missing context or overly generic phrasing, so we could target improvements where they mattered most.
To scale evaluation across development cycles, we introduced an LLM-as-a-Judge. This enabled us to compare different agent architectures and LLMs, and to continuously evaluate the impact of prompt tweaks.
Our dataset of real-world partner replies also helped benchmark embedding models for retrieval quality. Between MiniLM and E5-Small, MiniLM proved the most balanced and efficient default choice, particularly for English, while E5-Small performed better on non-English cases. These insights informed our multilingual optimization strategy, which combines language detection with embedding selection to maintain retrieval quality across locales.
For live monitoring, we used Arize, an AI observability platform that lets us inspect sampled traces, monitor agent behavior, and identify potential issues early. We also integrated in-tool user feedback and controlled experiments to validate model performance in production.
Each evaluation cycle delivered actionable insights that guided the next iteration, refining the agent’s logic, data usage, and overall reliability through a continuous feedback loop.
Impact and Future Directions
Today, the agent helps partners manage tens of thousands of guest messages every day, out of about 250,000 daily partner-guest exchanges. For supported topics, it can suggest or send a reply within minutes. In live pilots, this human-in-the-loop approach boosted user satisfaction by 70%, reduced follow-up messages, and sped up response times. Partners report spending less time on repetitive questions and seeing more consistent communication. When confidence is low or a message falls outside policy, the agent steps aside, keeping accuracy and trust at the center.
Building the agent has been a learning journey. One key takeaway has been the need to balance speed, cost and robustness. Complex agentic systems, especially those involving multi-step reasoning, can quickly become expensive in both latency and compute cost. We’ve learned that it’s crucial to think about efficiency from the very start, not as an afterthought. Our end-to-end evaluation pipeline, powered by an automated LLM-based judge, has been instrumental in helping us iterate quickly while maintaining high performance.
We’ve also learned that many guest messages are requests for action, not just questions, things like modifying bookings or arranging special services. Handling these well will require deeper contextual understanding and a longer-term partner memory, moving the agent from a smart responder toward a more capable operational assistant.
Looking ahead, we’re exploring personalization, adapting tone and style to match each partner’s unique voice. The goal isn’t just faster replies, it’s communication that feels effortless, natural and aligned with how our partners already engage with their guests.
